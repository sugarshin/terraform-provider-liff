# Release terraform-provider-liff

Cut and verify a new versioned release of this provider.

**Invocation:** run manually via `/release`. If the user passed an argument
(`$ARGUMENTS`) it selects the target version in step 2 — a bump keyword (`patch` /
`minor` / `major`) or an explicit version (`v0.0.5`). With no argument, infer the bump
from the diff and confirm. This skill is `disable-model-invocation` because it performs
an irreversible, outward-facing action (pushing a tag → public release → Registry
publish), so it should only ever run when the user explicitly asks for it.

## How releasing works here (mechanism)

A release is triggered **entirely by pushing a `vX.Y.Z` git tag** — there is no manual
publish step:

```
git push origin vX.Y.Z
      │
      ▼
.github/workflows/release.yml   (on: push tags 'v*')
      ├─ test  job    → go build + go test
      └─ release job  → GoReleaser (release --clean)
                         builds all os/arch zips, SHA256SUMS, GPG .sig, manifest.json
                         and creates the GitHub Release
      │
      ▼  (outside the workflow — Terraform Registry webhook)
Terraform Registry auto-ingests the new version → registry.terraform.io/providers/sugarshin/liff
```

Key facts (verified):
- The **GitHub Actions workflow does NOT call the Registry**. It only produces a
  correctly-formatted, GPG-signed GitHub Release. The Registry ingests it on its own,
  usually within a few minutes.
- **One-time setup is already done** (repo claimed on the Registry + GPG public key
  registered). `v0.0.1`–`v0.0.3` released and ingested successfully, so no setup work
  is needed for a new release.
- Provider address on the Registry is **`sugarshin/liff`** (namespace `sugarshin`,
  type `liff`).
- There is currently **no `CHANGELOG.md`** and GoReleaser's changelog is disabled
  (`.goreleaser.yml` → `changelog.disable: true`), so the GitHub Release body comes out
  empty. See "If a changelog is later added" below.

## Procedure

Work through these steps in order. **Steps 1–2 are read-only; step 4 (tag push) is the
irreversible, outward-facing action — always confirm with the user before it.**

### 1. Pre-flight checks (must all pass)

```bash
git fetch origin --tags
git rev-parse --abbrev-ref HEAD                 # expect: main
git status --porcelain                          # expect: empty (clean tree)
[ "$(git rev-parse HEAD)" = "$(git rev-parse origin/main)" ] && echo "in sync" || echo "NOT in sync"
gh run list --workflow test.yml --branch main --limit 1   # expect the latest to be: success
```

Release from `main`, fully in sync with `origin/main`, clean working tree, and with the
latest `main` Test run green. If any fails, stop and surface it — don't tag a dirty or
behind tree.

**Recommended local gates** (the release workflow only runs `go build` + `go test`, so
run these to catch what CI won't):

```bash
make test
make lint
make generate && git status --porcelain   # docs/ MUST stay clean; a diff means docs are
                                           # stale and must be committed BEFORE releasing
```

`docs/` is generated by `tfplugindocs`; CI fails if it's out of date, so a non-empty diff
here means you need a commit on `main` first, then restart from step 1.

### 2. Pick the next version

```bash
git describe --tags --abbrev=0                              # latest tag, e.g. v0.0.3
git log --oneline "$(git describe --tags --abbrev=0)"..main # what's shipping since then
```

**If `$ARGUMENTS` was given**, use it directly:
- an explicit version (`v0.0.5`) → use as-is (validate it's greater than the latest tag).
- a bump keyword (`patch` / `minor` / `major`) → apply it to the latest tag.

**Otherwise**, choose the SemVer bump from the diff and tell the user which and why:
- **patch** (`z+1`) — bug fixes, dependency bumps, CI/docs only (the common case here,
  since most changes are Renovate updates).
- **minor** (`y+1`, reset `z`) — new resource / data source / schema field (backward
  compatible).
- **major** (`x+1`, reset `y,z`) — breaking schema/behavior change. (Still `0.x` here, so
  breaking changes conventionally go in a minor bump until `1.0.0`.)

### 3. Confirm

Show the user: the exact version to tag, the target commit (`git rev-parse --short HEAD`),
and the one-line summary of what's shipping. Get an explicit go-ahead before proceeding —
pushing a tag kicks off a public release and Registry ingestion.

### 4. Tag and push (triggers the release)

```bash
git tag -a vX.Y.Z -m "vX.Y.Z"
git push origin vX.Y.Z
```

Use an **annotated** tag (`-a`) to match the existing `v0.0.1`–`v0.0.3` tags.

### 5. Watch CI

Give the run a couple of seconds to register, then:

```bash
gh run list --workflow release.yml --limit 1
gh run watch "$(gh run list --workflow release.yml --limit 1 --json databaseId -q '.[0].databaseId')" --exit-status
```

Both the `test` and `release` jobs must succeed.

### 6. Verify the release

```bash
gh release view vX.Y.Z
```

Confirm the assets include, for this version: the per-os/arch `..._{os}_{arch}.zip` files,
`terraform-provider-liff_X.Y.Z_SHA256SUMS`, its `..._SHA256SUMS.sig`, and
`terraform-provider-liff_X.Y.Z_manifest.json`. These are what the Registry requires.

Then confirm Registry ingestion (may lag a few minutes behind the GitHub Release):

```bash
curl -s -o /dev/null -w "%{http_code}\n" https://registry.terraform.io/v1/providers/sugarshin/liff/X.Y.Z
# 200 = ingested and live; 404 = not yet, wait and retry
```

Report the result: workflow status, release URL, and whether the Registry has the new
version live.

## If something goes wrong (rollback a bad tag)

If the workflow failed or the tag pointed at the wrong commit **and the Registry has NOT
yet ingested it**:

```bash
git push origin :refs/tags/vX.Y.Z    # delete the remote tag
git tag -d vX.Y.Z                    # delete the local tag
gh release delete vX.Y.Z --yes       # delete the GitHub Release if one was created
```

Then fix the underlying problem, and re-release.

**Important — Registry versions are immutable.** Once the Terraform Registry has ingested a
version number, it **cannot be re-published** under the same number. If a broken release
already reached the Registry, do **not** try to reuse the number — bump to the next patch
(e.g. `vX.Y.(Z+1)`) and release that instead.

## If a changelog is later added

If the project adopts a committed `CHANGELOG.md` (release-prep flow), insert a step
**between 3 and 4**: regenerate/confirm the changelog for the new version, merge that
change into `main` first, then tag the commit that includes it (so the tag captures the
changelog). If instead only GitHub Release notes are adopted (`.github/release.yml`
categories), no extra manual step is needed — the notes generate on release automatically.
